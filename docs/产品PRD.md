# 院内离线病历文书结构化采集与人工核验工作站 — 产品 PRD

## 1. 产品定位

本产品定位为**院内离线病历文书结构化采集与人工核验工作站**。

- **运行环境**：医生准备的私人 Windows 电脑（主机），手机仅作为拍照采集终端
- **核心流程**：手机拍摄电脑屏幕上的病历文书页面 → 图片与框选元数据上传到本地电脑 → 后端调用外部本地算法模块完成图像处理、文档解析与字段抽取 → 电脑端网页进行人工审核、修正、确认和导出
- **算法依赖说明**：图像预处理、OCR 文字识别、文档解析、LLM 结构化字段抽取等算法模块由外部团队独立开发交付。本系统将这些模块作为本地依赖后续直接打包集成，不通过远程API调用外部服务，保证后续的可演进性
- **边界约束**：不接入医院 HIS/EMR，不连接外部服务器，不调用云端接口，不自动生成诊断结论，不自动写回病历系统
- **数据传输**：手机与电脑之间的数据传输仅发生在本地局域网或电脑热点环境内，不经过公网服务器、即时通讯工具、网盘、邮箱或第三方中转服务

**核心价值**：把"人工逐字录入"转化为"电脑端人工核验与修正"。

---

## 2. 样例文档特征

目标文档来源为医院系统电脑屏幕上的病历文书页面，由手机拍屏采集。典型问题包括：屏幕摩尔纹、倾斜、反光、字体较小、页边灰色背景、页面跨页。

内容特征：
- 以中文长段落为主
- 包含标准病历结构：主诉、现病史、既往史、个人史、婚育史、家族史、体格检查、辅助检查、初步诊断、诊疗计划、签名、日期等
- 部分页面包含生命体征、检验指标、诊断列表、医生签名、时间信息和小型表格化内容

**处理策略**：按"多页病历文书"处理，而非"单张标准表格"。前端审核以电脑端为主。

---

## 3. 核心目标

1. **手机快速采集**：支持医生使用手机快速采集多页病历文书图片，自动归并为一个病历任务
2. **离线处理**：支持本地 Windows 电脑离线处理图片，形成可核验的文本结果和结构化字段结果
3. **电脑端人工审核**：在电脑端查看原图、解析文本和结构化字段，对字段进行快速确认、修改、补全和删除
4. **结构化导出**：优先支持 Excel（人工流转）和 JSON（系统集成/数据留存）
5. **简单部署**：不要求 Docker、WSL、GPU 驱动、开发环境或云服务客户端

---

## 4. 非目标范围（第一版）

- 不做医院系统接口对接，不写回 HIS/EMR
- 不自动诊断，不生成医学建议，不进行疾病风险预测
- 不做患者身份管理系统
- 不做多医生协同平台
- 不做云端同步
- 不做移动端复杂审核

**第一版策略**：追求"自动生成初稿 + 人工核验确认"，最终可信结果以人工确认后的结果为准。

---

## 5. 用户角色

| 角色 | 职责 |
|------|------|
| 医生/病案整理人员 | 拍摄病历页面、查看自动抽取结果、修正错误字段、确认结构化结果、导出 |
| 系统维护者 | Windows 离线部署包准备、模型与配置文件放置、启动脚本配置、日志查看和问题排查 |

手机端用户与电脑端用户通常是同一人：手机端只负责采集，电脑端负责审核。

## 6. 业务主流程

```
启动本地工作站
    ↓
电脑端点击"新建采集" → 创建采集会话 + 创建病历任务（状态："采集中"）
    ↓
电脑端生成采集二维码
    ↓
手机扫码进入采集页
    ↓
拍摄病历文书页面
    ↓
手动调整四边形目标范围（框选病历区域）
    ↓
确认框选范围
    ↓
上传当前页面
    ↓
继续采集下一页 或 完成采集
    ↓
完成采集后任务进入"上传完成"
    ↓
调用外部本地算法模块解析文本与字段
    ↓
电脑端人工核验
    ↓
确认并导出结构化结果
```

**关键环节 — 文档区域框选**：手机端拍摄后，系统显示可调整的四边形框选范围。用户通过拖拽四个角点手动调整识别范围，将识别区域限定在病历页面主体内，排除屏幕灰色背景、工具栏、页边空白和非病历内容。确认后，前端上传原始图像、用户确认的四边形坐标、页序和图片尺寸。框选完全由用户手动调节，本项目不实现自动边界识别算法；裁剪、透视矫正等图像处理由外部本地算法模块完成。

**分支流程 — 采集会话管理**：

- **取消采集**：用户可在完成采集前主动取消采集会话，已上传页面进入清理流程，关联任务标记为"失败"。
- **修订采集**：完成采集后若在电脑端发现漏拍、图片质量不佳、页序错误或处理失败，用户可点击"修订采集"解锁已锁定会话，回到编辑态后重新新增、删除、补拍、重新框选或调整页序。修订采集只需要一次确认防误触，不要求填写原因；解锁后关联任务回到"采集中"状态，重新完成采集后再次触发处理。修订采集不改变会话 ID 和任务 ID。
- **重新处理**：待审核任务可点击"重新处理"重新调用算法模块（适用于算法模块更新或怀疑处理异常时），任务回到"处理中"状态。

**采集会话过期时间**：默认 30 分钟，过期后会话不可继续上传。电脑端工作台可在"新建采集"时修改过期时间。

## 7. 前端产品需求

### PR-FE-001：电脑端工作台

电脑端工作台首页是系统主入口，主要用于展示系统运行状态、发起手机采集、查看当前采集会话、处理待办任务和进入最近任务。首页面向医生/病案整理人员，不直接展示本机访问地址、局域网访问地址、端口号等技术信息；相关连接信息仅在“系统状态”或“帮助中心”的高级说明中展示。

首页应包含以下内容：

- 系统状态区：展示系统名称、离线运行状态、系统启动状态和帮助中心入口。状态表达应采用医生可理解的业务语言，例如“系统已启动”“离线运行”“手机采集可用”，避免展示 IP、端口等技术信息。
- 新建采集入口：首页核心操作为”新建采集”。用户点击后，系统同时创建采集会话和病历任务（状态为”采集中”），并以弹窗形式展示手机扫码二维码。弹窗中可设置会话过期时间（默认 30 分钟）。二维码仅在用户主动新建采集或查看当前会话时显示，不常驻首页。
- 扫码采集弹窗：弹窗展示采集二维码、会话状态、剩余时间、已上传页数，并提供”重新生成二维码””关闭””手机无法连接？”等操作。手机需与电脑处于同一网络环境，相关说明以帮助提示形式展示。
- 当前采集会话卡片：当存在进行中的采集会话时，首页展示当前会话状态，包括会话是否进行中、已上传页数、已上传页面缩略图预览、剩余时间，并提供”查看二维码””取消采集”和”完成采集”操作。已上传页面缩略图以横向滚动的形式展示，点击可放大查看完整图片。无进行中会话时展示空状态，引导用户新建采集。
- 已锁定会话卡片：当会话已完成采集时，展示”已锁定”状态、已上传页面缩略图预览，并提供”修订采集”操作。确认后会话回到编辑态，可重新上传、删除、调整页面。
- 任务概览区：展示关键任务状态统计，包括”待审核””处理中””采集中””处理失败””已导出”。点击对应卡片后进入任务列表并按状态筛选。
- 最近任务列表：展示最近任务的任务编号、页面缩略图、创建时间、页数、当前状态、审核进度和操作按钮。任务编号旁展示首页缩略图。操作按钮应随任务状态变化，包括”开始审核””查看进度””查看原因””重新处理””修订采集””导出结果””查看结果”等。
- 系统提醒区：展示最近的关键提醒，包括处理失败、手机连接异常、导出成功等。提醒文案应清晰说明问题和下一步操作，避免使用开发者错误信息。
- 桌面端导航一致性：工作台总览、任务管理、人工审核、导出等电脑端一级或任务内页面应保持同一工作站外壳。侧边栏必须持续提供“工作台总览”和“任务管理”入口，并明确当前所在模块；点击任务管理、任务状态统计或任务操作不得跳转到缺少返回路径的孤立界面。

**验收标准**：
双击启动脚本后浏览器自动打开电脑端工作台首页。
首页显示系统已启动、离线运行等业务化状态信息，不直接展示 IP、端口、本机访问地址或局域网访问地址。
点击”新建采集”后同时创建采集会话和病历任务（状态为”采集中”），并弹窗展示二维码；弹窗中可修改会话过期时间。
二维码按需展示，关闭弹窗后不常驻首页。
当前存在采集会话时，首页可查看已上传页数、已上传页面缩略图、剩余时间，并支持重新打开二维码、取消采集或完成采集。
已锁定会话支持”修订采集”，确认后回到编辑态可重新上传、删除、调整页序；任务处理中时不可修订采集。
任务概览能够正确展示待审核、处理中、采集中、处理失败和已导出数量。
最近任务列表每项展示任务编号、首页缩略图、创建时间、页数、状态和对应操作（含”修订采集””重新处理”等）。
从首页进入任务管理后，页面仍保留同一工作站侧边栏，可直接切回工作台总览；桌面端模块切换不得丢失主导航。
任务为空时显示明确空状态，并引导用户新建采集。
手机连接、任务处理、导出等异常应有明确提示，并提供可恢复操作。

### PR-FE-002：手机端文档采集页

负责病历页面的拍摄、文档区域框选、页面预览、重拍、上传和多页采集。**不承担**复杂字段审核、结构化结果展示、最终确认。

核心操作流程：
1. 扫码进入采集会话
2. 点击单个"拍摄/选择图片"入口，调用手机浏览器的拍照或相册选择能力
3. 进入框选页，系统默认给出覆盖图片中心约 80% 的四边形目标范围
4. 仅通过拖动四个角点调整识别范围，适配屏幕倾斜、跨页显示等场景
5. 确认上传当前页面，或在上传前重拍
6. 上传成功后回到已采集页面列表；用户可继续采集下一页，直至点击"完成采集"
7. 完成采集前可在页面列表中删除、拖拽调整页序、补拍或重新框选已上传页面

手机端视觉应对齐 `docs/UI_image/目标设计稿/` 下的三类目标图：未拍照态、框选态、已采集列表态。白底顶栏包含返回、标题和帮助入口；顶部展示绿色"采集会话进行中"状态胶囊和一行简短说明，避免堆叠长段说明。未拍照态主卡片使用浅蓝弱背景、页面计数和单个"拍摄/选择图片"入口；零页时页面列表展示空状态，"完成采集"禁用并提示至少上传 1 页。框选态必须在一个手机视口内完成主要操作，图片预览、浅蓝提示条、"将上传原图与框选坐标"说明和"重拍/重新框选/确认上传"按钮都应可见，不依赖向下滚动寻找关键操作。已采集列表态采用行型卡片：左侧显示页面缩略图，中部显示页序、上传状态和上传时间，右侧显示删除操作；页面之间支持拖拽排序（长按拖拽），并在底部提供补拍页面、提示条、继续拍下一页和完成采集。已上传页面缩略图应展示用户框选后的四边形区域预览；前端只基于用户已确认的四边形坐标生成前端预览效果，不做 OCR、透视矫正或算法裁剪。页面布局应在 360px~428px 宽度的主流手机视口下均可正常使用。

四边形框选交互应按照固定四角语义处理：用户必须从某个可拖动圆点开始拖动，拖动期间只移动该圆点；点击图片或线条空白处不得把最近圆点瞬移到点击位置。拖动应使用 pointer capture 或等效机制保持连续跟手，限制在图片边界内，并在会导致四边形自相交时阻止提交或保持有效形状。框选页不得出现三角形、交叉线或角点顺序错乱。

支持的操作：拍摄/选择图片、手动调整四边形框选范围、重拍、确认上传、继续拍下一页、完成采集、重试上传失败页、删除页面、拖拽排序调整页序、补拍页面、重新框选已上传页面。四边形框选完全由用户手动调节，前端不实现自动边界识别算法。

**验收标准**：手机浏览器中可完成多页采集；拍摄和相册选择均可触发设备能力；每页上传前可调整四边形识别范围，且不展示像素值输入、坐标滑块等开发者控件；四角拖动连续跟手，不因触碰空白区域导致角点跳动或四边形交叉；上传请求包含原图、框选坐标、页序和图片尺寸；已上传页面可重新框选并保存新的框选坐标；页面列表展示框选后的四边形区域预览；多页可通过拖拽排序调整顺序，并按当前列表顺序归入同一任务；上传失败有明确提示并可只重试失败页；框选页关键内容保持在一个手机视口内，底部操作不遮挡页面列表；在 360px~428px 宽度主流手机视口下均可正常操作；不要求安装 App。

### PR-FE-003：任务列表

电脑端展示所有病历任务，至少显示：任务编号、首页缩略图、创建时间、页数、处理状态、审核状态、导出状态。

任务状态流转：采集中 → 上传完成 → 处理中 → 待审核 → 已确认 → 已导出。处理中若图像处理、文档解析或字段抽取模块未配置、调用失败、返回空结构化结果或返回结构不符合约定，任务进入"失败"分支并展示明确错误原因，不做规则兜底或人工降级处理。待审核任务可"重新处理"回到处理中；上传完成、待审核或失败任务可通过"修订采集"回到采集中。

**验收标准**：新建采集后任务列表自动新增（状态为"采集中"）；状态随处理进度更新；处理失败可查看原因并重新处理；任务列表每项展示首页缩略图；待审核任务可重新处理；符合条件的已锁定任务可修订采集；任务列表作为电脑端工作站模块打开，必须保留工作站侧边栏和返回工作台总览的入口。

### PR-FE-004：电脑端人工审核

审核页面核心目标是**降低医生核对成本**。用户可同时查看原始图片、解析文本和结构化字段，并能快速修改字段值。

- 支持多页切换：按页查看原图，也可查看合并后的文本和字段结果
- 每个字段可编辑、清空、确认、标记存疑
- 支持批量确认：用户可勾选多个字段后一键确认，提升审核效率
- 字段失焦时自动保存，但仅在字段值发生变化时才触发保存请求。保存中应显示明确状态，保存失败时保留用户输入并提示重试，不得静默丢弃修改。
- 系统保存人工修改痕迹

**验收标准**：电脑端完成全部字段审核；字段可编辑；修改后失焦自动保存（仅值变化时触发）；保存中、保存成功、保存失败状态可见；保存失败不清空用户输入且允许重试；刷新后已保存修改不丢失；可批量勾选字段并一键确认；未确认任务不可标记为最终完成。

### PR-FE-005：字段来源核验

每个结构化字段应尽量关联来源文本。展示字段时支持查看来源 OCR 文本或来源页面。无来源的字段标记为"未定位来源"或"需人工确认"。

**验收标准**：点击字段可查看对应来源文本；无来源字段提示用户人工核验；导出前提示仍未确认的字段数量。

### PR-FE-006：字段状态管理

每个字段有独立状态，用于前端提醒和导出前校验。

字段状态至少包括：未审核、已确认、已修改、存疑、为空、空值已确认。

- **为空**：字段值为空且尚未人工确认。导出前在预警面板中提示。
- **空值已确认**：用户已显式确认"空值可接受"。常见场景包括未婚患者的"婚育史"、无过敏史患者的"过敏史"等。该状态等同于已确认，不阻断导出。

**验收标准**：各状态转换正确；`empty` 字段可显式确认为"空值可接受"，状态变为 `confirmed_empty`；导出前可统计各状态字段数量；`confirmed_empty` 不记入阻断项。

### PR-FE-007：导出功能

支持导出当前任务的结构化结果，第一版支持 Excel 和 JSON。

- Excel：用于人工查看
- JSON：用于后续系统集成或数据处理
- 导出内容必须为人工审核后的最终结果，而非自动抽取原值
- 导出文件包含任务编号、导出时间、字段结果和审核状态

导出前系统展示完整性检查面板，列出未审核字段数、存疑字段数、为空字段数、未定位来源字段数和空值已确认字段数。存在未审核或存疑字段时，系统展示预警提示并说明风险，用户确认后仍可继续导出。导出文件中保留字段审核状态。

**验收标准**：可导出 Excel 和 JSON；导出内容与审核后字段一致；导出前展示完整性检查面板；存在未审核/存疑字段时展示预警但允许用户知情后继续导出；导出失败有明确错误提示。

### PR-FE-008：错误提示与可恢复性

对所有常见错误进行明确提示，包括：手机无法连接电脑、图片上传失败、图片过大、任务处理失败、字段抽取失败、导出失败等。

可恢复操作包括：
- 上传失败可重试当前失败页
- 处理失败可重新处理
- 会话锁定后可在符合条件时修订采集
- 采集进行中可取消会话
- 导出失败不影响已审核数据

**验收标准**：任一流程失败时用户能看到可理解的错误信息；上传失败可重试；处理失败可重新处理；符合条件的已锁定会话可通过修订采集回到编辑态；进行中会话可取消；导出失败不影响已审核数据。

### PR-FE-009：手机端文档框选与采集元数据

用户拍摄或选择病历页面后，系统在照片上叠加一个四边形目标范围用于标记有效识别区域。默认框选范围覆盖图片中心约 80% 区域，用户不调整也可直接确认上传。框选完全由用户手动调节，本项目不实现自动边界识别算法。

- 支持四个角点拖动
- 用户确认后上传原始图像和框选元数据
- 已上传页面支持重新进入框选页并更新该页的框选元数据
- 框选页只展示图片、四边形 overlay、四个角点、重拍/确认操作和必要提示，不展示像素坐标输入、滑块或裁剪效果伪预览
- 主要解决：页面倾斜、背景干扰、跨页内容、工具栏遮挡、页边空白过多等问题

**验收标准**：拍照或选择图片后可见四边形框选范围；默认范围可直接确认；可拖动角点调整；确认上传时请求包含用户确认的框选范围；对已上传页面重新框选后，新的框选坐标可保存到对应页面；不调整时使用默认范围；框选组件异常时提示错误并允许重新拍摄。

## 8. 后端产品需求

### PR-BE-001：本地服务启动

后端以本地服务形式运行在 Windows 电脑上，启动后提供前端页面、API 接口、文件上传接口和任务处理能力。必须支持离线运行：不请求外部网络资源，不依赖云端 API，不联网下载模型或依赖包。

**验收标准**：断网下系统可启动；浏览器可访问本地工作台；手机在同一局域网或电脑热点下可访问采集页。

### PR-BE-002：采集会话管理

每次点击"新建采集"时，系统同时创建采集会话和病历任务（状态为"采集中"）。二维码对应一个上传会话，手机扫码进入后，上传的多张图片归入同一个病历任务。会话包含唯一标识、关联任务 ID、创建时间、过期时间（默认 30 分钟，可修改）、访问地址、上传页数和状态。

采集会话在用户点击"完成采集"前处于 `active` 可编辑状态，允许新增、删除、调整页序、补拍页面和重新框选已上传页面。用户点击"完成采集"后，会话进入 `locked` 状态，关联任务进入 `uploaded`。锁定后的页面顺序和框选元数据作为文档合并、图像处理和字段抽取的默认输入。

**会话取消**：`active` 状态的会话可由用户主动取消，进入 `cancelled` 终态。取消后已上传页面进入清理流程，关联任务进入 `failed`。

**修订采集**：`locked` 状态的会话在关联任务不是 `processing` 时，可由用户在电脑端确认后解锁回到 `active` 编辑态。解锁后关联任务回到 `capturing` 状态，用户可重新新增、删除、补拍、重新框选和拖拽调整页序。重新完成采集后，任务再次进入 `uploaded → processing → ready_for_review`。修订采集不改变会话 ID 和任务 ID；不要求填写原因。`processing` 状态不可修订采集，避免处理过程中同时修改输入。

**会话过期**：超过过期时间后会话进入 `expired` 终态。若有已上传页面，关联任务保留为 `capturing`；若无已上传页面，关联任务进入 `failed`。

完成采集前，后端应允许前端更新已上传页面的框选坐标；完成采集后，新增、删除、排序、补拍和重新框选均不可用，除非电脑端进入修订采集流程。

**验收标准**：每次点击"新建采集"同时创建会话和任务（`capturing`）；同二维码上传的多页图片归入同一任务；会话过期（默认 30 分钟）后不可继续上传，过期时间可修改；`active` 会话可取消，进入 `cancelled` 并清理页面；非 `processing` 的 `locked` 会话可通过修订采集回到 `active`，任务回到 `capturing`；修订采集不改变会话 ID 和任务 ID；点击"完成采集"前会话处于可编辑状态；点击"完成采集"后会话锁定且页面顺序和框选元数据固化。

### PR-BE-003：图片上传与文件管理

后端接收手机上传的图片，保存到本地任务目录。每个任务有独立目录，避免文件混淆。

文件存储按任务隔离，分为上传目录（原始图片、框选元数据、外部模块成功返回的处理后图片路径）、结果目录（解析结果和审核结果）、导出目录（生成的 Excel 和 JSON 文件）。

**验收标准**：上传图片不覆盖其他任务文件；多页按顺序保存；可返回图片访问路径；删除任务时可清理对应文件。

### PR-BE-004：任务生命周期管理

以任务为核心组织业务流程。每个任务在"新建采集"时即创建（状态为"采集中"），随采集、处理、审核、确认、导出阶段流转。

任务状态包括：采集中、上传完成、处理中、待审核、已确认、已导出、失败。

- **采集中**：任务已创建，采集会话活跃，页面可新增、删除、排序、补拍、重新框选。
- **上传完成**：采集完成，会话已锁定，页序和框选元数据固化，可触发算法处理。
- **处理中**：外部算法模块正在处理中。
- **待审核**：外部算法模块已成功生成候选字段，等待医生核验。可点击"重新处理"回到处理中。
- **已确认**：人工审核已确认，可导出；若个人使用中发现采集错误，也可修订采集。
- **已导出**：已导出最终结果；若个人使用中发现采集错误，也可修订采集后重新处理和导出。
- **失败**：处理失败或会话异常终止，可重新处理，也可修订采集。

文档解析或字段抽取不完整、失败、未配置或契约非法时进入"失败"，系统保留错误原因和已产生的中间结果用于排查，但不允许后端基于规则或 schema 生成替代字段。

**验收标准**：新建采集时任务以 `capturing` 状态创建；状态可查询且变更有记录；`ready_for_review` 可回到 `processing`（重新处理）；符合条件的 `locked` 会话修订采集后任务回到 `capturing`；失败任务保留错误信息并可重新处理；解析或抽取不完整时进入"失败"状态并明确报错，不直接崩溃，也不进入人工降级流程。

### PR-BE-005：文档解析服务集成

文档解析能力（图像预处理 + OCR）由外部团队交付的算法模块提供，本系统将其打包为本地依赖直接调用。后端 service 层负责将任务图片传入模块、接收解析结果并持久化，API 层仅触发任务。

外部算法模块输入输出约定（待外部团队交付后细化）：

- **输入**：任务图片列表（外部图像处理模块成功返回的处理后图像路径；若未生成则按模块契约报错）
- **输出**：统一文档解析结果，包含逐页纯文本、段落块、表格块、页码信息
- **错误处理**：部分页面解析失败时保留成功页面结果，标记失败页

**验收标准**：给定一组图片可调用算法模块生成解析结果；算法模块不可用、调用失败或返回无效结果时任务进入失败状态并有明确错误提示。

### PR-BE-006：结构化字段抽取服务集成

字段抽取能力（LLM）由外部团队交付的算法模块提供，本系统将其打包为本地依赖直接调用。后端 service 层将文档解析结果与字段 schema 传入模块，接收结构化字段候选结果并持久化。

外部算法模块输入输出约定（待外部团队交付后细化）：

- **输入**：文档解析结果 + 字段 schema 定义
- **输出**：结构化字段候选结果，包含字段名、字段值、来源证据、置信度
- **约束**：该模块只生成候选字段，最终结果必须经过人工审核

**验收标准**：可调用算法模块根据 schema 生成结构化字段；抽取失败、返回空字段候选或返回结构不符合 schema 时任务进入"失败"状态并明确报错。后端不得基于规则、模板或 schema 自行抽取/补造字段。

### PR-BE-007：字段 Schema 管理

支持配置结构化字段 schema。第一版使用固定 schema 文件，后续可扩展前端管理。

第一版 schema 覆盖以下字段组：

| 字段组 | 包含字段 |
|--------|----------|
| 患者基本信息 | 姓名、性别、年龄、ID号、住院号、病区、床号、科室 |
| 入院/病程信息 | 入院时间、记录时间、主诉、现病史、既往史、个人史、婚育史、家族史 |
| 体格检查 | 体温、脉搏、呼吸、血压、身高、体重、BMI、一般情况、专科体征 |
| 辅助检查 | 检查日期、检查类型、检查所见、检查结论、关键检验指标 |
| 诊断相关 | 初步诊断、最后诊断、修正诊断、诊断列表 |
| 治疗相关 | 主要治疗、用药信息、处理意见、诊疗计划 |
| 签名与时间 | 记录医生、上级医生、签名时间、出院/入院日期 |

Schema 包含版本号，每个任务记录所使用的 Schema 版本，避免模板调整后影响历史任务。

Schema 应支持文书类型标识，例如 admission_record、first_course_record、discharge_record、exam_report 等。第一版可使用通用病历 Schema，后续可按文书类型扩展不同字段模板。

**验收标准**：修改 schema 文件后抽取字段范围随之变化；schema 变更不需要修改前端核心代码；前端能根据 schema 动态展示字段组。

### PR-BE-008：人工审核结果保存

审核结果与自动抽取结果分开保存，避免覆盖原始候选结果。系统保留三类数据：自动结果、人工结果、修改痕迹。

支持单字段保存和批量字段确认。批量确认时前端提交字段 key 列表，后端一次性更新对应字段状态为 `confirmed`。

**验收标准**：修改字段后保存人工结果；支持批量确认字段（一次 API 调用更新多个字段状态）；再次打开任务显示人工修改后的内容；仍可追溯自动抽取原值。

### PR-BE-009：导出服务

基于人工审核结果生成导出文件。第一版支持 Excel（按字段组组织）和 JSON。

导出前系统进行完整性检查，返回未审核字段数、存疑字段数、为空字段数、未定位来源字段数和空值已确认字段数。存在未审核或存疑字段时，导出接口返回预警信息（不阻断），前端展示预警面板后由用户确认是否继续导出。导出文件中保留字段审核状态。

**验收标准**：导出前进行完整性检查并返回统计数据；存在未审核/存疑字段时返回预警但不阻断；导出内容来自人工审核结果（最终值）；导出文件保存到本地指定目录；导出文件包含字段审核状态。

### PR-BE-010：本地日志与错误追踪

记录系统启动、图片上传、任务处理、字段抽取、人工审核、导出等关键事件。日志仅保存在本地，不上传。日志包含任务编号和错误原因，但不记录过多敏感原文以避免隐私泄露。

**验收标准**：出现错误时可在日志目录找到对应记录；日志包含任务编号和错误原因。

### PR-BE-011：页面裁剪结果与采集元数据保存

后端接收手机端上传的页面图像及其框选元数据。对每页保存：原始图像、用户确认的四边形坐标、外部图像处理模块成功返回的处理后图像路径、页序号、上传时间和框选更新时间。

- 外部图像处理模块成功返回的处理后图像作为默认文档解析输入，同时保留原始图像用于追溯和排查
- 若手机端未提供框选坐标，按普通上传流程处理，不阻断任务创建
- 完成采集前，允许手机端更新已上传页面的四边形坐标；更新坐标不得要求重新上传原图，也不得改变页序
- 支持配置原始图像与处理后图像的保留策略，允许用户导出完成后手动清理

**验收标准**：可接收带框选坐标的上传；可在会话锁定前更新已上传页面的框选坐标；能区分原始图像和外部模块返回的处理后图像；文档解析默认使用外部模块成功返回的处理后图像；任务详情中可查看每页原图和处理后图像路径；缺少框选信息时任务仍能正常创建；外部图像处理失败时任务处理进入失败状态并明确报错。

## 9. 核心数据实体

| 实体 | 核心属性 |
|------|----------|
| **Task（任务）** | 任务标识、当前状态（采集中/上传完成/处理中/待审核/已确认/已导出/失败）、来源会话、创建/更新时间、页数、异常信息 |
| **CaptureSession（采集会话）** | 会话标识、关联任务 ID、状态（active/locked/expired/cancelled）、过期时间（默认 30 分钟）、局域网地址、已上传页数、创建/锁定/取消时间 |
| **PageImage（页面图像）** | 所属任务、页序号、原始图像路径、处理后图像路径、尺寸、四边形角点坐标、上传时间、框选更新时间 |
| **DocumentResult（文档解析结果）** | 所属任务、逐页结果（纯文本、段落块、表格块）、合并全文文本、创建时间 |
| **StructuredField（结构化字段）** | 字段标识、字段名、自动抽取值、来源证据文本、来源页码、审核状态、置信度 |
| **ReviewResult（审核结果）** | 所属任务、审核总体状态、逐字段审核记录（最终值、原值、审核状态、审核时间） |

系统同时保留自动抽取原值和人工修改后的最终值，支持完整追溯。

## 10. API 设计概要

API 按职责分为以下模块：

| 模块 | 主要能力 |
|------|----------|
| **系统与会话** | 查询系统状态、创建采集会话（同时创建任务）、查询会话信息、取消会话、修订采集/解锁会话、修改过期时间、生成采集二维码 |
| **手机上传** | 上传原始图像、上传带框选坐标的页面、更新已上传页面框选坐标、完成采集、查询会话状态 |
| **任务管理** | 任务列表、任务详情、触发处理、重试、重新处理、修订采集、删除任务 |
| **文档结果** | 获取页面图像列表（含缩略图）、获取文档解析结果、获取结构化字段结果 |
| **人工审核** | 获取审核结果、修改单个字段、批量确认字段、确认任务、标记存疑、确认空值可接受 |
| **导出** | 导出完整性检查（返回预警统计）、导出 Excel、导出 JSON |

## 11. 前后端职责边界

| 层 | 职责 | 不负责 |
|----|------|--------|
| **前端** | 采集交互、页面展示、字段编辑、确认操作、导出触发 | OCR、字段抽取、文件系统细节 |
| **后端** | 文件接收、任务状态、调用外部本地算法模块、结果持久化、导出生成 | 复杂页面交互、写死字段展示逻辑、OCR/LLM/图像处理算法实现 |
| **算法模块（外部交付）** | 图像预处理、OCR、文档解析、LLM 字段抽取，由外部团队独立开发交付后打包集成 | — |

前后端通过 API 解耦。算法模块作为本地依赖打包集成，替换或升级算法模块不影响前端 API 和核心业务流程。
