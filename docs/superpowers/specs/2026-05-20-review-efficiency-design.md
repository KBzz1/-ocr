# 审核页效率优化设计

## 背景

当前 MVP 已打通工作台新建任务、手机多图上传、后端模拟处理、电脑端审核、完成和导出。审核页已有基础三栏界面：任务图片、OCR 文本、结构化字段，支持字段编辑、保存、标记完成和导出。

下一步目标不是重新实现审核页，而是把现有审核页从“能完成流程”优化为“医生能快速连续核验”的工作界面。

## 目标

- 提高医生核验字段的速度，减少视线和鼠标移动。
- 多页任务中能快速定位当前页图片和字段来源；OCR 文本作为可选辅助信息，不强迫医生持续查看。
- 字段列表优先支持快速浏览和批量统一审核，避免逐字段点击确认。
- 保持 MVP 简单字段状态：`unreviewed / confirmed / modified`。
- 保持仓库边界：前端只展示后端返回的图片、OCR 文本、字段候选和来源证据，不从 schema、OCR 文本或页面内容推断、补造字段。

## 非目标

- 不新增 `suspicious / empty / confirmed_empty` 等复杂字段状态。
- 不做字段自动抽取、规则兜底、OCR 文本解析或 schema 推断。
- 不做四边形框选、裁剪、透视矫正、图片增强或来源框高亮。
- 不新增采集会话、修订采集、拖拽排序或补拍替换。
- 不实现新的导出格式或导出前复杂完整性预警面板。

## 推荐方案

采用“统一审核工作台”方案：以图片和结构化字段为主工作区，OCR 文本作为可折叠辅助栏。

- 左栏：页缩略导航和当前页原图，默认只展开当前页大图。
- 中栏：结构化字段表单，支持快速浏览、直接修改和统一审核。
- 右侧或下方抽屉：OCR 文本，可折叠，默认收起或保持用户上次选择。
- 顶部：任务摘要、修改数量、保存状态、统一审核完成操作。

这个方案改动集中在 `ReviewPage`、`FieldList`、`ReviewSourcePanel` 和 CSS，复用现有 API，不要求后端新增契约，适合 MVP 后的第一轮体验打磨。

备选方案 A 是保留三栏常驻 OCR，对照充分但页面拥挤，且医生不看 OCR 时浪费屏幕。备选方案 B 是逐字段确认模式，质控颗粒度更细，但日常审核成本高。当前不采用。

## 页面结构

### 顶部任务栏

顶部固定在页面内容上方，不覆盖内容。展示：

- 任务编号。
- 当前任务状态。
- 页数。
- 字段摘要：字段总数、已修改数量、仍未统一审核数量。
- 保存状态：未保存修改、保存中、已保存、保存失败。
- 主要操作：保存、统一审核并完成、导出 JSON、导出 Excel。
- 辅助操作：显示/隐藏 OCR。

统一审核并完成时，前端把仍处于 `unreviewed` 且未被医生修改的字段统一标记为 `confirmed`，把被医生修改的字段保留为 `modified`，再保存并请求完成任务。这个动作代表医生对当前字段表单做整体确认，不要求逐字段点击。

如果保存或完成接口返回 `REVIEW_NOT_COMPLETED` 或其他校验错误，页面展示后端错误文案，并保留当前编辑内容。

### 左栏：页导航和原图

左栏用于快速切换页面。

- 显示页码列表，页码来自 `review_result.pages`。
- 默认选中第一页。
- 点击页码后切换当前页原图；如果 OCR 栏处于展开状态，同步切换当前页 OCR 文本。
- 当前页有 `preview_url` 优先展示 `preview_url`，否则展示 `image_url`。
- 图片缺失时展示“后端未返回当前页原图”，不崩溃。
- 多页图片不再全部大图纵向展开，避免审核页过长。

### OCR 辅助栏

OCR 文本是辅助信息，不是默认必看区域。

- 默认收起 OCR 栏，或读取本地 UI 状态保持用户上次选择。
- 顶部提供“显示 OCR / 隐藏 OCR”按钮。
- 展开后默认展示当前页 `parsed_text`。
- 如果当前页没有 `parsed_text`，展示 `review_result.ocr_text` 或“后端未返回 OCR 文本”。
- 提供“当前页 / 合并文本”切换；合并文本只使用后端返回的 `review_result.ocr_text` 或 pages 中已有文本拼接，不做内容推断。
- 当用户选中字段且字段 evidence 包含文本片段时，若 OCR 栏展开，则在 OCR 文本中高亮该片段。
- evidence 文本找不到时，展示“来源文本未在当前 OCR 中定位”，不自行搜索其它字段或生成来源。

### 中栏：字段表单

字段表单是主要工作区。

- 字段顺序完全沿用后端返回顺序。
- 每个字段展示字段名、状态、当前人工值、自动候选值和来源页。
- 输入框默认使用 `field.value`；归一化逻辑继续由 `api/review.ts` 保证 `value/final_value/candidate_value` 的兼容。
- 修改字段值后状态立即变为 `modified`，但只在用户保存后提交后端。
- 不提供逐字段“确认”按钮。
- 字段卡片支持点击后成为当前焦点字段，联动 OCR 辅助栏来源提示和左栏页码。
- 空字段列表展示“后端未返回可审核字段”，不提供人工新增字段入口。

统一审核规则：

- 用户没有改动的 `unreviewed` 字段，在“统一审核并完成”时统一标记为 `confirmed`。
- 用户改动过的字段保持 `modified`。
- 已经是 `confirmed` 或 `modified` 的字段保持原状态，除非用户再次修改字段值。
- 统一审核动作不新增字段、不删除字段、不改写自动候选值。

## 快捷操作

第一轮只实现不会改变业务边界的快捷操作：

- `Ctrl+S` / `Cmd+S`：保存当前审核结果。
- `Enter`：在字段输入框内保留默认输入行为，不提交整页。
- `Ctrl+Enter` / `Cmd+Enter`：触发“统一审核并完成”。如果存在未保存修改，先保存，再完成。

暂不实现复杂键盘流，如方向键跨字段、自动跳转下一字段。后续如要做，需要单独补充可用性设计。

## 数据流

加载：

1. `GET /api/tasks/{taskId}/review`。
2. 页面保存 `status`、`review_result`、`fields`、`selectedPageId`、`selectedFieldKey`。
3. `selectedPageId` 默认为第一条 page 的 `page_id`。

编辑：

1. 用户修改字段值。
2. 前端仅更新本地 `fields`，字段状态改为 `modified`。
3. 页面进入“未保存修改”状态。

确认：

1. 用户点击“统一审核并完成”或按 `Ctrl+Enter` / `Cmd+Enter`。
2. 前端把仍为 `unreviewed` 的未修改字段统一改为 `confirmed`。
3. 修改过的字段保持 `modified`。
4. 页面进入“保存中”状态，并执行保存与完成流程。

保存：

1. 用户点击保存或按 `Ctrl+S` / `Cmd+S`。
2. `PUT /api/tasks/{taskId}/review` 提交当前字段数组。
3. 成功后以后端返回的 `review_result.fields` 替换本地字段。
4. 失败时保留本地编辑内容，展示错误，允许再次保存。

完成：

1. 统一审核动作先调用 `PUT /api/tasks/{taskId}/review` 保存整体字段状态和值。
2. 保存成功后调用 `POST /api/tasks/{taskId}/complete`。
3. 成功后任务状态进入 `done`，导出保持可用。
4. 任一步失败时展示后端错误，不清空未保存编辑。

导出：

- 继续复用 `ExportPanel`。
- `review` 和 `done` 状态可导出。
- 导出失败只展示错误，不影响审核字段。

## 错误处理

- 审核数据加载失败：显示错误面板，不渲染可编辑字段。
- failed 任务直接访问审核页：以后端 `INVALID_TASK_TRANSITION` 为准，展示“审核数据加载失败”及后端错误原因，不显示人工补字段入口。
- 保存失败：保留用户输入，显示“保存失败，请重试”或后端错误文案。
- 完成失败：保留当前页面和编辑内容，显示后端错误文案。
- 图片加载失败：当前页图片区域显示占位文案，不影响字段审核。

所有错误文案不得包含完整病历原文、身份证号、图片 base64、模型输出全文或堆栈。

## 测试策略

组件测试：

- 审核页默认选中第一页，只展示当前页大图。
- OCR 辅助栏默认收起；点击“显示 OCR”后展示当前页 OCR 文本。
- 点击页码切换当前图片；OCR 展开时同步切换当前页 OCR 文本。
- 字段修改后状态变为 `modified`，页面显示未保存状态。
- 点击“统一审核并完成”后，未修改字段统一提交为 `confirmed`，已修改字段提交为 `modified`。
- 保存失败保留本地编辑内容并允许重试。
- OCR 展开时，字段 evidence 文本存在则高亮；不存在时显示来源不可定位提示。
- 空字段列表不出现人工新增字段入口。

E2E 测试：

- MVP 成功流继续通过：新建任务、上传 3 张图、完成上传、进入审核、修改字段、保存、完成、导出。
- 新增审核效率流：多页 review mock 下切换页码、隐藏/显示 OCR、修改字段、统一审核完成，所有 API 请求必须被 mock。
- 失败直达审核页继续不渲染可编辑字段。

验证命令：

conda run -n manzufei_ocr python -m pytest app/backend/tests -q
npm --prefix app/frontend test
env -i HOME=$HOME PATH=/usr/bin:/bin npm --prefix app/frontend run build

## 实施切片

1. 审核页状态模型：补充 `selectedPageId`、`selectedFieldKey`、`isOcrVisible`、dirty/save status 和派生字段摘要。
2. 页导航和当前页原图：替换全部图片纵向展开为页码导航加当前页预览。
3. OCR 折叠辅助栏：显示/隐藏 OCR，展开后支持当前页/合并文本切换和 evidence 高亮。
4. 字段列表效率优化：字段卡片增加来源页、自动候选值、状态显示和焦点联动，移除逐字段确认入口。
5. 统一审核完成：未修改字段批量标记 confirmed，修改字段保持 modified，保存成功后完成任务。
6. 保存体验：保存按钮状态、快捷键保存、失败保留编辑内容。
7. 测试补齐：组件测试、E2E 审核效率流、现有成功/失败 E2E 回归。

## 成功标准

- 医生能在一个页面内以图片和字段为主完成快速审核，OCR 可按需展开。
- 常见操作路径为：浏览字段 → 修改少数字段 → 统一审核并完成 → 导出。
- 多页任务不会因为全部图片纵向展开导致页面过长。
- 审核页不要求医生逐字段点击确认。
- 审核页不出现人工新增字段、人工降级、规则兜底或前端推断字段行为。
- 后端和前端现有 MVP E2E 仍全部通过。
