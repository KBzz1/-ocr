# 审核页效率优化设计

## 背景

当前 MVP 已打通工作台新建任务、手机多图上传、后端模拟处理、电脑端审核、完成和导出。审核页已有基础三栏界面：任务图片、OCR 文本、结构化字段，支持字段编辑、保存、标记完成和导出。

下一步目标不是重新实现审核页，而是把现有审核页从“能完成流程”优化为“医生能快速连续核验”的工作界面。

## 目标

- 提高医生核验字段的速度，减少视线和鼠标移动。
- 多页任务中能快速定位当前页图片、当前页 OCR 文本和字段来源。
- 字段列表能清楚区分未审核、已确认、已修改，支持快速确认和保存。
- 保持 MVP 简单字段状态：`unreviewed / confirmed / modified`。
- 保持仓库边界：前端只展示后端返回的图片、OCR 文本、字段候选和来源证据，不从 schema、OCR 文本或页面内容推断、补造字段。

## 非目标

- 不新增 `suspicious / empty / confirmed_empty` 等复杂字段状态。
- 不做字段自动抽取、规则兜底、OCR 文本解析或 schema 推断。
- 不做四边形框选、裁剪、透视矫正、图片增强或来源框高亮。
- 不新增采集会话、修订采集、拖拽排序或补拍替换。
- 不实现新的导出格式或导出前复杂完整性预警面板。

## 推荐方案

采用“审核工作台增强”方案：保留现有三栏信息架构，但优化为更高密度、更少跳转的工作台。

- 左栏：页缩略导航和当前页原图，默认只展开当前页大图。
- 中栏：当前页 OCR 文本，提供“当前页 / 合并文本”切换。
- 右栏：字段核验列表，按字段逐项编辑、确认、保存。
- 顶部：任务摘要、审核进度、保存状态、完成操作。

这个方案改动集中在 `ReviewPage`、`FieldList`、`ReviewSourcePanel` 和 CSS，复用现有 API，不要求后端新增契约，适合 MVP 后的第一轮体验打磨。

备选方案 A 是做“字段优先单栏表格”，效率高但原图和 OCR 对照弱，容易降低核验质量。备选方案 B 是做完整质控工作台，包含批量确认、来源缺失统计和导出预警，但会提前引入后置复杂能力。当前不采用。

## 页面结构

### 顶部任务栏

顶部固定在页面内容上方，不覆盖内容。展示：

- 任务编号。
- 当前任务状态。
- 页数。
- 字段审核进度：未审核、已确认、已修改数量。
- 保存状态：未保存修改、保存中、已保存、保存失败。
- 主要操作：保存、标记完成、导出 JSON、导出 Excel。

标记完成前不在前端强制阻止未审核字段，但如果后端返回 `REVIEW_NOT_COMPLETED` 或其他校验错误，页面展示后端错误文案，并保留当前编辑内容。

### 左栏：页导航和原图

左栏用于快速切换页面。

- 显示页码列表，页码来自 `review_result.pages`。
- 默认选中第一页。
- 点击页码后切换当前页原图和当前页 OCR 文本。
- 当前页有 `preview_url` 优先展示 `preview_url`，否则展示 `image_url`。
- 图片缺失时展示“后端未返回当前页原图”，不崩溃。
- 多页图片不再全部大图纵向展开，避免审核页过长。

### 中栏：OCR 文本

中栏用于对照文本。

- 默认展示当前页 `parsed_text`。
- 如果当前页没有 `parsed_text`，展示 `review_result.ocr_text` 或“后端未返回 OCR 文本”。
- 提供“当前页 / 合并文本”切换；合并文本只使用后端返回的 `review_result.ocr_text` 或 pages 中已有文本拼接，不做内容推断。
- 当用户选中字段且字段 evidence 包含文本片段时，在 OCR 文本中高亮该片段。
- evidence 文本找不到时，展示“来源文本未在当前 OCR 中定位”，不自行搜索其它字段或生成来源。

### 右栏：字段核验列表

右栏是主要工作区。

- 字段顺序完全沿用后端返回顺序。
- 每个字段展示字段名、状态、当前人工值、自动候选值和来源页。
- 输入框默认使用 `field.value`；归一化逻辑继续由 `api/review.ts` 保证 `value/final_value/candidate_value` 的兼容。
- 修改字段值后状态立即变为 `modified`，但只在用户保存后提交后端。
- 点击“确认”将字段状态改为 `confirmed`，保留字段值不变。
- 字段卡片支持点击后成为当前焦点字段，联动中栏来源提示和左栏页码。
- 空字段列表展示“后端未返回可审核字段”，不提供人工新增字段入口。

## 快捷操作

第一轮只实现不会改变业务边界的快捷操作：

- `Ctrl+S` / `Cmd+S`：保存当前审核结果。
- `Enter`：在字段输入框内保留默认输入行为，不提交整页。
- 字段确认按钮保持显式点击，避免误确认。

暂不实现复杂键盘流，如方向键跨字段、批量确认、自动跳转下一字段。后续如要做，需要单独补充可用性设计。

## 数据流

加载：

1. `GET /api/tasks/{taskId}/review`。
2. 页面保存 `status`、`review_result`、`fields`、`selectedPageId`、`selectedFieldKey`。
3. `selectedPageId` 默认为第一条 page 的 `page_id`。

编辑：

1. 用户修改字段值。
2. 前端仅更新本地 `fields`，字段状态改为 `modified`。
3. 页面进入“未保存修改”状态。

确认：

1. 用户点击字段“确认”。
2. 前端仅更新本地字段状态为 `confirmed`。
3. 页面进入“未保存修改”状态。

保存：

1. 用户点击保存或按 `Ctrl+S` / `Cmd+S`。
2. `PUT /api/tasks/{taskId}/review` 提交当前字段数组。
3. 成功后以后端返回的 `review_result.fields` 替换本地字段。
4. 失败时保留本地编辑内容，展示错误，允许再次保存。

完成：

1. 用户点击“标记完成”。
2. `POST /api/tasks/{taskId}/complete`。
3. 成功后任务状态进入 `done`，导出保持可用。
4. 失败时展示后端错误，不清空未保存编辑。

导出：

- 继续复用 `ExportPanel`。
- `review` 和 `done` 状态可导出。
- 导出失败只展示错误，不影响审核字段。

## 错误处理

- 审核数据加载失败：显示错误面板，不渲染可编辑字段。
- failed 任务直接访问审核页：以后端 `INVALID_TASK_TRANSITION` 为准，展示“审核数据加载失败”及后端错误原因，不显示人工补字段入口。
- 保存失败：保留用户输入，显示“保存失败，请重试”或后端错误文案。
- 完成失败：保留当前页面和编辑内容，显示后端错误文案。
- 图片加载失败：当前页图片区域显示占位文案，不影响字段审核。

所有错误文案不得包含完整病历原文、身份证号、图片 base64、模型输出全文或堆栈。

## 测试策略

组件测试：

- 审核页默认选中第一页，只展示当前页大图。
- 点击页码切换当前图片和当前页 OCR 文本。
- 字段修改后状态变为 `modified`，页面显示未保存状态。
- 字段确认后状态变为 `confirmed`，保存请求包含更新后的字段数组。
- 保存失败保留本地编辑内容并允许重试。
- 字段 evidence 文本存在时 OCR 文本高亮；不存在时显示来源不可定位提示。
- 空字段列表不出现人工新增字段入口。

E2E 测试：

- MVP 成功流继续通过：新建任务、上传 3 张图、完成上传、进入审核、修改字段、保存、完成、导出。
- 新增审核效率流：多页 review mock 下切换页码、修改字段、确认字段、保存，所有 API 请求必须被 mock。
- 失败直达审核页继续不渲染可编辑字段。

验证命令：

```bash
conda run -n manzufei_ocr python -m pytest app/backend/tests -q
npm --prefix app/frontend test
env -i HOME=$HOME PATH=/usr/bin:/bin npm --prefix app/frontend run build
env -i HOME=$HOME PATH=/usr/bin:/bin npm --prefix app/frontend run test:e2e
```

说明：当前默认 Node 可能是 24.x，Playwright E2E 需要 Node 18/20/22；使用系统 Node 18 的 `PATH=/usr/bin:/bin` 运行。

## 实施切片

1. 审核页状态模型：补充 `selectedPageId`、`selectedFieldKey`、dirty/save status 和派生审核进度。
2. 页导航和当前页原图：替换全部图片纵向展开为页码导航加当前页预览。
3. OCR 文本切换和 evidence 联动：当前页/合并文本切换，字段选中后显示来源提示和文本高亮。
4. 字段列表效率优化：字段卡片增加来源页、自动候选值、确认状态和焦点联动。
5. 保存体验：保存按钮状态、快捷键保存、失败保留编辑内容。
6. 测试补齐：组件测试、E2E 审核效率流、现有成功/失败 E2E 回归。

## 成功标准

- 医生能在一个页面内完成多页图片、OCR 文本和结构化字段的快速对照。
- 常见操作路径为：选字段 → 对照来源 → 修改或确认 → 保存 → 完成。
- 多页任务不会因为全部图片纵向展开导致页面过长。
- 审核页不出现人工新增字段、人工降级、规则兜底或前端推断字段行为。
- 后端和前端现有 MVP E2E 仍全部通过。
