# 病历文书结构化采集与人工核验工作站 — 前端 TDD 测试用例（优化版）

> 基于 `docs/产品prd.md` 的 PR-FE-001 ~ PR-FE-009 编写。本文档是前端实现前的 TDD 测试清单，目标是让测试覆盖业务闭环，同时严格守住新版 PRD 的职责边界。

## 0. 审查结论与边界修正

原测试用例覆盖了工作台、手机采集、任务列表、审核、字段状态、导出和错误处理，方向是合格的；但需要修正以下问题：

- 过多用例仍在验证实际图像处理效果，例如“预览已排除灰色背景、工具栏”“透视矫正后的像素结果”。新版 PRD 中图像处理交给外部团队，本项必须改为只测前端交互、坐标采集和上传契约。
- 不应测试 OCR/LLM 语义正确性。前端只展示后端成功返回的解析文本、字段候选、来源证据和字段状态，不判断文本识别或字段抽取是否正确。
- 状态枚举缺少 `created`、`uploading` 等 PRD 状态，需补齐并统一中文标签；算法模块未配置、异常或结构化抽取失败时必须显示 `failed`，不得进入人工降级流程。
- 多页采集缺少“完成采集前可编辑，完成后锁定”的关键用例。
- 缺少手动选择局域网地址、无外部网络请求、隐私日志不外泄、未 mock 请求失败等质量门禁。
- 原文档大量测试只有注释，没有明确的失败原因、mock 数据和断言。实现时每个用例必须落成可执行测试，并先跑出 RED。

## 1. 测试总原则

### 1.1 本项目前端负责

- 本地工作台展示、二维码与采集入口展示
- 手机端拍照/选择图片、四边形框选交互、页序管理、上传状态
- 任务列表、状态流转展示、失败重试入口
- 审核页原图/文本/字段展示、字段编辑、状态管理、来源核验
- 导出触发、导出前校验、错误提示和恢复路径

### 1.2 本项目前端不负责

- OCR 文字识别准确率
- LLM 字段抽取准确率
- 自动识别文档边界的算法正确性
- 透视矫正、裁剪、去摩尔纹、去反光等图像处理效果
- 医学诊断、医学建议或字段医学合理性判断

涉及上述能力时，前端测试只验证：

- 是否调用本地后端 API 或注入的外部模块适配器
- 外部模块不可用、异常或返回无效结果时是否显示明确错误
- 后端返回成功结果中的空字段、无来源字段时页面是否可审核；但算法失败或未配置时不得进入审核页

## 3. 统一状态与契约

### 3.1 任务状态

| 后端值 | 前端标签 | 关键断言 |
|--------|----------|----------|
| `created` | 已创建 | 可继续上传或等待页面 |
| `uploading` | 上传中 | 显示上传进度或忙碌态 |
| `uploaded` | 上传完成 | 可进入处理或等待处理 |
| `processing` | 处理中 | 禁止确认和导出 |
| `ready_for_review` | 待审核 | 可进入审核页 |
| `confirmed` | 已确认 | 可导出 |
| `exported` | 已导出 | 显示导出记录 |
| `failed` | 失败 | 显示原因和重新处理入口 |

### 3.2 字段状态

| 后端值 | 前端标签 | 导出前处理 |
|--------|----------|------------|
| `unreviewed` | 未审核 | 统计并提示 |
| `confirmed` | 已确认 | 允许确认/导出 |
| `modified` | 已修改 | 允许确认/导出 |
| `suspicious` | 存疑 | 统计并重点提示 |
| `empty` | 为空 | 统计并提示，可人工确认 |

### 3.3 算法错误返回

当前项目中 OCR/LLM/图像处理模块未配置、异常或返回无效结构时，后端必须把任务置为 `failed`。前端测试必须覆盖以下失败返回：

```ts
export const algorithmFailedTask = {
  task_id: 'task_20260511_0001',
  status: 'failed',
  error_code: 'ALGORITHM_MODULE_NOT_CONFIGURED',
  error_message: '算法模块未配置，无法生成结构化字段',
};
```

前端必须显示错误原因和“重新处理”入口，不得显示“人工补录后继续确认/导出”的降级路径。

## 4. TDD 质量门禁

每个用例落地时必须满足：

- 测试名描述一个可观察行为，避免 `works`、`test1` 之类命名。
- 先写测试并运行到 RED，失败原因应是功能缺失或断言不满足，而不是 import/语法错误。
- mock 数据必须是最小必要数据，不通过大而全 fixture 掩盖断言。
- 对异步 UI 使用 `findBy*`、`waitFor` 或 Playwright expect，不使用固定 `sleep`。
- 所有外部请求、CDN 字体、远程图片请求必须在测试中失败。
- 算法相关用例只测“外部成功返回/错误提示”，不测实际算法结果；算法未配置、异常、结构化字段为空或契约非法时必须按失败处理。

| FE-ERR-007 | 组件 | 重复提交、重复上传、重复导出被前端按钮忙碌态阻止 |
| FE-ERR-008 | E2E | 任一流程失败后刷新页面，可从后端状态恢复到一致 UI |

## 15. 离线、安全与隐私

| ID | 类型 | 用例 |
|----|------|------|
| FE-SEC-001 | E2E | 页面加载时无 CDN、远程字体、远程图片或外部 API 请求 |
| FE-SEC-002 | E2E | `context.setOffline(true)` 下，已本地加载页面仍可使用 MSW mock 完成主要流程 |
| FE-SEC-003 | 组件 | 错误 toast 不展示完整病历原文、身份证号等敏感内容 |
| FE-SEC-004 | 组件 | 文件名中含路径穿越字符时，前端只展示安全文件名，不拼接成本地路径 |
| FE-SEC-005 | E2E | 控制台不输出完整 OCR 文本、字段原文或图片 base64 |
| FE-SEC-006 | 组件 | 未 mock API 请求必须让测试失败，防止误接真实网络 |

## 16. 端到端关键路径

### FE-E2E-001：完整采集到人工导出

步骤：

1. 电脑端打开工作台，显示系统状态和空任务列表。
2. 新建采集会话，显示二维码和局域网访问地址。
3. 移动端通过 session URL 进入采集页。
4. 选择 3 张本地 fixture 图片。
5. 每页调整四边形角点，只断言坐标变化和上传请求体，不断言图像处理效果。
6. 完成采集后会话锁定。
7. 电脑端任务列表出现新任务，状态流转到 `ready_for_review`。
8. 打开审核页，展示后端成功返回的解析文本和结构化字段候选。
9. 修改字段，确认若干字段，标记一个字段存疑。
10. 确认审核先因存疑字段被阻断；处理存疑并确认全部字段后，任务进入 `confirmed`。
11. 导出 JSON 和 Excel，验证触发下载和导出记录。

### FE-E2E-002：算法模块失败必须报错

后端返回 `ALGORITHM_MODULE_NOT_CONFIGURED` 时：

- 任务状态为 `failed`。
- 任务列表和详情页显示错误原因。
- 审核页入口不可用或进入后只显示失败态。
- 页面提供重新处理入口。
- 不出现“人工补录后继续确认/导出”的降级路径。

### FE-E2E-003：离线约束

- 页面不请求外部域名。
- 所有 mock API 使用本地地址。
- 断网不导致前端加载远程资源失败。

## 18. 建议实施顺序

1. 工作台基础状态、二维码、空列表。
2. 手机采集会话、文件选择、上传状态。
3. 四边形框选 UI 与坐标上传契约。
4. 页面管理、完成采集、锁定会话。
5. 任务列表状态展示和重试。
6. 算法失败态、审核页成功结果、人工编辑保存。
7. 字段状态、来源提示、确认校验。
8. 导出触发和错误恢复。
9. 离线、安全、E2E 主流程。

每一轮必须按 RED → GREEN → REFACTOR 执行。算法能力只能来自外部成功 fixture；未配置、异常、空结构化结果或契约非法都按失败处理，不允许前端或后端补一套降级逻辑。
